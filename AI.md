# AI Guide for umbriel

Everything below is reachable from `umbriel` (or its unscoped alias `umbriel`) alone. **Bun + Windows only.** This file is the complete surface ÔÇö an agent should not need to read the source. When it must, the "Where to look" section says exactly which one file to open.

## What it is

Playwright for Windows desktop apps. Query the live UI Automation (accessibility) tree by name / control-type / automationId, invoke controls, type, read values, wait for elements to appear, and serialize a window's tree to JSON for an LLM agent. Three accessibility engines:

1. **IUIAutomation COM client** (the spine) ÔÇö driven through a cast-free vtable invoker. Query + invoke + patterns + cache.
2. **`oleacc` MSAA fallback** ÔÇö for legacy / owner-draw windows that expose no useful UIA tree.
3. **Java Access Bridge** (`jab.ts` ÔÇö top-level `javaTree` / `javaInvoke` / `javaSetText`) ÔÇö reads and drives Java Swing/AWT (and bridged JavaFX) windows, which expose NOTHING to UIA or MSAA.

Escalation rule: stay on the `umbriel` facade. Drop to a lower engine (`msaaTree`, the raw `vcall`) only when you need something the facade lacks.

**More than accessibility.** The same facade drives synthetic input (`type`/`click`/`sendKeys` + raw mouse/keyboard helpers), cursor-free interaction (`invoke`/`setValue`/`postClick` ÔÇö no real-cursor movement, works locked), full-screen + per-window screenshots (with a background **Windows.Graphics.Capture** path), image **template matching** and **OCR** (Windows.Media.Ocr) for surfaces with no a11y tree, **clipboard** read/write (text / image / files), native **HWND introspection** (Spy++/Winspector-style), an LLM **computer-use** adapter (Anthropic/OpenAI action sets), and a zero-dep **MCP server** ÔÇö one package covering the a11y (FlaUI), pixel (nut.js/robotjs), and window-spy (Spy++) niches at once.

## Mental model (read this first)

- **UIA is cross-process.** Every property read and `find` marshals into the target app. `initialize()` once (it sets up a single-threaded COM apartment + makes the process DPI-aware); `uninitialize()` at the end, or `using app = umbriel.attach(...)`.
- **Selectors are hybrid.** Exact scalars (`controlType`, `name` string, `automationId`, `className`) compile to a **server-side** UIA condition ÔÇö the target app filters in-process and only matches come back. Rich predicates (`name` RegExp, `nameContains`) are matched **client-side** on the (already-narrowed) results. Always scope a search to a window (`attach`), **never** `findAll` from the desktop root.
- **CacheRequest batches.** Naive walks pay N cross-process round-trips; `findAllCached` / `tree()` prefetch a whole subtree in one. The cache wins more the larger the tree.
- **Only the listed patterns are proven.** `invoke`, `value`/`setValue`, `text`, `toggle`, `expand`/`collapse`, `select`, `rangeValue`/`setRangeValue`, container `scroll`/`setScrollPercent`/`scrollInfo` (ScrollPattern ÔÇö proven on File Explorer's items view, 0%ÔåÆ100%), `move`/`resize`/`rotate` + `canMove`/`canResize`/`canRotate` (TransformPattern ÔÇö proven on Notepad: move+resize shifted its boundingRectangle exactly, cursor-free), window `close`/`setVisualState` are each proven against a real control. `scrollIntoView` (ScrollItemPattern) is implemented.
- **Synthetic input needs an unlocked, interactive desktop.** `type`, `sendKeys`, `click` go through `SendInput` ÔÇö they are silently dropped on a locked session. UIA queries, `invoke`, and `setValue` all work locked (`screenshot`/PrintWindow ÔÇö and the WGC fallback ÔÇö can come back blank when the session is locked). Prefer `setValue` / `invoke` over `type` / `click` when a pattern exists.
- **Element pointers have apartment affinity.** Keep them on the creating thread; tolerate `UIA_E_ELEMENTNOTAVAILABLE (0x80040201)` if a window closes mid-walk. Window/process lifecycle **events ship** (`waitForWindow` via `SetWinEventHook`; `waitForProcess` polls a toolhelp32 snapshot); UIA property/structure event SUBSCRIPTION is out of scope (MTA) ÔÇö poll control state with `waitFor`.

## Full API

### `umbriel` ÔÇö the facade object
`attach(target)`, `launch(command, target, timeout?)`, `focused()`, `fromPoint(x, y)`, `elementAt(x, y)`, `root()`, `windows()`, `tree(element, options?)`, `snapshot(window, options?)`, `diff(before, after)`, `waitForIdle(element, options?)`, `execute(element, actions)`, `msaaTree(hWnd, maxDepth?)`, `windowTree(hWnd, maxDepth?)`, `click(x, y)`, `postClick(x, y, button?)`, `postKey(hWnd, key)`, `postText(hWnd, text)`, `setControlText(hWnd, text)`, `isKeyDown(name)`, `scrollAt(x, y, dir, amount?)`, `sendKeys(combo)`, `type(text)`, `captureScreen(region?)`, `screenshotScreen(region?)`, `captureWindowLive(hWnd, options?)`, `pixelColor(x, y)`, `listMonitors()`, `locateOnScreen(needle, options?)`, `readClipboard()`, `readClipboardFiles()`, `readClipboardImage()`, `writeClipboard(text)`, `writeClipboardFiles(paths, move?)`, `writeClipboardImage(image)`, `paste(text)`, `copy()`, `dispatch(window, action, options?)` (computer-use), `listProcesses()`, `killProcess(pid)` ÔåÆ 'killed'|'denied'|'not-found' (TerminateProcess, no taskkill; MCP `kill_process`), `systemResources(sampleMs?)` ÔåÆ `SystemResources` (RAM/CPU/uptime/process-count; MCP `system_resources`), `waitForWindow(match, options?)`, `waitForWindowGone(match, options?)`, `waitForProcess(imageName, options?)`, `waitForProcessGone(imageName, options?)`, `watchWindows(handler, options?)`, `ocrWindow(hWnd, options?)`, `ocrScreen(region?, options?)`, `ocrBitmap(bitmap, options?)`, `initialize()`, `uninitialize()`. (`ocrAvailable()` and `disposeOcr()` are bare named exports, not `umbriel.` facade methods.)

**Drive in the dark (the doctrine):** an AI is not a human at a screen ÔÇö it does not need a window foregrounded, visible, under a cursor, or on the active desktop. Cursor-free / no-foreground is the **default**: `invoke` / `setValue` / `toggle` / `scroll`, an element-targeted posted click (the MCP `click`/`find_and_act` resolve the control's OWN window via `ownerHwnd` ÔåÆ `postClickToHwnd`), and the posted key/text path (`setControlText` / `postKey` / `postText` / `pasteToControl` ÔåÆ a control's `nativeWindowHandle`, behind the MCP `type` (WM_CHAR) and `paste` (WM_PASTE) tools when the target has its own HWND) act on a minimized, background, occluded, or locked window with no focus theft and no real cursor. (This holds for a classic **Win32/HWND** app; a **UWP/WinUI** store app SUSPENDS its UI thread + a11y tree when minimized or fully backgrounded ÔÇö its tree then reads empty and posted actions may not land until you `restoreWindow`/`raiseWindow` it. A backgrounded-but-not-minimized, or occluded, window is usually still live.) (A bare coordinate `umbriel.postClick(x, y)` posts to whatever window is TOPMOST at that pixel ÔÇö occlusion-correct only when the pixel belongs to your target; prefer the element/ref path when a window may overlap.) `captureWindowLive` SEES such a window; the whole UIA/native/MSAA tree of a background window is readable untouched. SendInput (`click` / `sendKeys` / `drag`, plus the `type`/`paste` fallback for a control with no own HWND) and `PrintWindow` are for (a) a human watching or (b) a pixel-only / no-own-HWND surface ÔÇö and the MCP server's `UMBRIEL_CURSOR=never` refuses every SendInput path (`drag`; the real-cursor `{cursor:true}` variant of `click_point`/`click_text`; `press_key` chord; `hold_key` WITHOUT an own-HWND `{ref}`; bare-`copy`; and the `type`/`paste`/`find_and_act`/`reveal` `{do:'type'}` SendInput fallback for a no-own-HWND control), while the cursor-free paths stay live: the DEFAULT posted `click_point`/`click_text` (so OCRÔåÆ`click_point` on a pixel-only surface still works), `type` (WM_CHAR) / `paste` (WM_PASTE) / `press_key {ref}` / `hold_key {ref}` (posted WM_KEYDOWN autorepeat) on an own-HWND control, `set_value`, and `copy {ref}`.

### `class Element`
- Live properties (getters): `name`, `controlType`, `controlTypeName`, `automationId`, `className`, `isEnabled`, `boundingRectangle: Rect`, `nativeWindowHandle: bigint`, `clickablePoint: {x,y} | null` (UIA GetClickablePoint), `value`, `toggleState`, `expandCollapseState`, `isSelected`, `isOffscreen` (scrolled/clipped out of view), `isPassword` (UIA IsPassword ÔÇö withhold `value` before emitting it anywhere a model/log sees it), `rangeValue`, `scrollInfo: ScrollInfo | null`.
- Tree: `find(selector, scope?)`, `findAll(selector, scope?)`, `findAllCached(selector, request, scope?)`, `children`, `parent`, `await waitFor(selector, { timeout?, interval? })`, `await waitForGone(selector, { timeout?, interval? })` (the inverse ÔÇö resolves when the selector no longer matches: the spinner / "LoadingÔÇĞ" / modal-dismissed gate; MCP `wait_for {gone:true}`), `await waitForState(selector, expectation: StateExpectation)` (the desktop `expect(locator).toBeChecked()/toHaveValue()` ÔÇö poll the matched control until its STATE matches `expectation` (or it times out) to CONFIRM an action landed: a toggle is now on, a set value stuck, an item is now selected/expanded/enabled. Re-finds the control every poll (so it survives a re-render), reads the same live getters the snapshot does, compiles the condition ONCE and never leaks a poll's Element. Returns the matched `Element` (caller releases) on success; THROWS on timeout quoting the last-seen state; MCP `wait_for {state:{toggle/expanded/selected/enabled/value/valueContains, ÔÇĞ}}`), `await waitForOwnState(expectation)` (the by-REF variant ÔÇö polls THIS exact control's live state instead of re-finding by selector, so a control with no unique selector can be awaited; probes liveness each poll and THROWS if the control was destroyed; MCP `wait_for {ref, state}`), `reveal(selector, { container?, maxSteps?, fromTop? })` (scroll a virtualized / off-screen item into the tree), `describeNoMatch(selector)`, `buildUpdatedCache(request)`, `cachedChildren`, `cached{Name,ControlType,AutomationId,ClassName,BoundingRectangle,IsEnabled}`.
- Reads (return null/empty if unsupported): `readTable(maxRows?, startRow?)` ÔåÆ `TableData { headers, rows, totalRows, startRow } | null` (GridPattern data grids / details lists / tables, cell-by-cell, column headers via TablePattern; `startRow` pages forward without re-reading from row 0); `cell(row, column)` ÔåÆ the cell Element of a Grid (compose setValue/invoke/toggle for a cursor-free cell edit); `getProperty(propertyId)` ÔåÆ `string | number | boolean | null` (ANY UIA property via GetCurrentPropertyValue ÔÇö `PropertyId.HelpText`/`IsOffscreen`/`HasKeyboardFocus`/`ItemStatus`/`FrameworkId`/ÔÇĞ ÔÇö the VARIANT is decoded by its vt tag and freed).
- Selection (SelectionItem/Selection patterns ÔÇö cursor-free on WinUI/WPF/Chromium; a classic own-HWND item routes through the bridge and moves foreground, disclosed by the MCP select tool): `select()` (replace), `addToSelection()` / `removeFromSelection()` (multi-select / deselect), `getSelection()` ÔåÆ owned `Element[]` (a container's selected items), `canSelectMultiple` getter.
- Text (TextPattern, cursor-free): `selectText(text, { backward?, ignoreCase? })` ÔåÆ matched text | null (the desktop getByText ÔÇö finds a substring and SELECTS it so you can copy/replace/read it), `getSelectedText()` ÔåÆ the current selection; `text()` ÔåÆ the full document text, `visibleText()` ÔåÆ only the on-screen ranges (GetVisibleRanges ÔÇö the bounded read for a huge terminal / editor; the terminal-scrollback path is proven in `example/terminal-scrollback.integration.test.ts`, where a console's `visibleText()` is a non-empty subset far smaller than its full `text()` scrollback).
- Pixels (occlusion-correct, cursor-free, background): `await capture()` ÔåÆ `Bitmap | null` ┬À `await ocr(options?)` ÔåÆ `OcrText | null` ÔÇö capture / OCR JUST this control's pixels (FlaUI `Capture.Element` / Playwright `locator.screenshot` parity), the see-pixels read for a `<canvas>`/chart/WebGL/custom-draw control with no a11y. Sources the element's OWN top-level window via WGC (`captureWindowLive` ÔÇö occluded/background/GPU-composited), falling back to PrintWindow (`captureWindowRGB`), then crops to this element's window-local bounds ÔÇö so an overlapping window's pixels can NEVER leak in (unlike `captureScreen(el.boundingRectangle)`, which blits whatever is TOPMOST at those screen pixels ÔÇö proven in `example/element-capture-occlusion.integration.test.ts`: `el.ocr()` reads the marker behind a maximized Explorer while the screen-region grab reads the occluder). `ocr()`'s per-word/line boxes are SCREEN-absolute (postClick a box centre). Null if the owner window has no surface (minimized / protected / off-screen). MCP `ocr {ref}` / `capture_window {ref}`.
- Patterns (throw if unsupported): `invoke()`, `doDefaultAction()` (MSAA LegacyIAccessible default action ÔÇö a cursor-free activate fallback), `setValue(text)`, `text()`, `toggle()`, `expand()`, `collapse()`, `select()`, `scrollIntoView()`, `scroll(horizontalAmount, verticalAmount)` + `setScrollPercent(h%, v%)` (container ScrollPattern ÔÇö cursor-free, works locked), `move(x, y)` + `resize(width, height)` + `rotate(degrees)` with `canMove`/`canResize`/`canRotate` (TransformPattern ÔÇö cursor-free geometry on an HWND-less child a top-level SetWindowPos cannot reach; FlaUI `Patterns.Transform` parity), `setRangeValue(n)`, `close()`, `setVisualState(WindowVisualState)`. Best-effort: `showContextMenu(): boolean` (IUIAutomationElement3 ÔÇö opens the control's context menu cursor-free; provider-dependent, the menu is an untitled popup to attach; false if unsupported). View modes: `views(): { current, supported: [{id,name}] } | null` ┬À `setView(id): boolean` (MultipleViewPattern ÔÇö proven on File Explorer's Items View, 8 named views, cursor-free).
- Input ÔÇö these `Element` methods take the SendInput fallback path and so need an unlocked, interactive session: `focus()` (UIA SetFocus is itself cursor-free, but the keystrokes you then send are not), `type(text)`, `click()`, `dragTo(target)`. `dragTo(target)` is the elementÔåÆelement drag (Playwright `locator.dragTo(target)` / FlaUI `element.DragAndDrop(target)`): it resolves THIS element's clickable point (bounding-rectangle center fallback, same as `click()`) and the target's, then runs the real-mouse `dragTo(fromX,fromY,toX,toY)` ÔÇö so an agent never hand-resolves the destination's pixels. REAL-MOUSE only (an OLE/HTML5 drop needs the live cursor); the MCP `drag` tool exposes it as `{ref, toRef}` (toRef's center computed for you), with `{toX,toY}` still the explicit-pixel alternative. For text/keys WITHOUT focus or an unlocked session, use the **cursor-free posted-message path** instead ÔÇö `umbriel.setControlText(hWnd, text)` (WM_SETTEXT) / `umbriel.postText(hWnd, text)` (WM_CHAR) / `umbriel.postKey(hWnd, key)` (WM_KEYDOWN/UP) / `pasteToControl(hWnd)` (WM_PASTE) to a control's own `nativeWindowHandle`, and the MCP `type` / `set_value` / `paste` / `press_key {ref}` tools, which all work on a MINIMIZED / background / occluded / locked own-HWND control with no focus theft (proven in `example/cursor-free-input.integration.test.ts`, which asserts the target is NOT the foreground window before each op). "Need an unlocked session" applies only to those SendInput fallbacks (raw `click()`/`type()`, and the `type`/`paste` fallback for a sub-control with no own HWND).
- Lifecycle: `release()`, `ptr: bigint`.

### `class Window extends Element`
Adds `hWnd: bigint`, `activate()`, `screenshot(): Uint8Array`, `webRoots(): Element[]` (the Chromium/Edge/Electron page-DOM render-widget roots ÔÇö splice into a snapshot via `snapshot(window, { extraRoots })` to read/act on web content; empty for non-Chromium windows; caller releases them), `dispose()` / `[Symbol.dispose]`.

### Selector & matching
`interface Selector { controlType?, controlTypes? ((ControlType|number)[]), name? (string | RegExp), nameContains?, automationId?, className?, index?, last? }`. `controlTypes` is a server-side OR ÔÇö match the clickable thing whether it's a Button OR a Hyperlink OR a MenuItem. `index` picks the Nth (0-based) match ÔÇö disambiguate N identical twins (three "Delete" buttons, 20 unnamed ListItems) without a snapshot round-trip; out-of-range yields no match. `last: true` picks the final match (sugar for the highest index); `index` wins when both are set. `interface ElementProperties { name, controlType, automationId, className }` (what `matches` reads). `matches(props, selector)`, `selectorToString(selector)`, `formatNoMatch(selector, windowName, candidateNames)`, `pickIndexed(candidates, selector)` (apply a selector's `index`/`last` to a candidate list ÔÇö the picker behind those two fields). `interface StateExpectation { enabled?, expanded?, selected?, toggle?, value? (exact ValuePattern match), valueContains? (substring), timeout? (ms, default 5000), interval? (ms, default 100) }` ÔÇö the expectation `waitForState` polls against; every predicate field is optional and the wait succeeds when ALL the provided ones hold at once, so set at least one (an all-absent expectation is vacuously true on the first poll).

### Root accessors (return a live `Element`/`Window`)
`fromHandle(hWnd)` (an `Element` for a window handle ÔÇö `attach` wraps this in a `Window`), `focused()`, `fromPoint(x, y)`, `root()`.

- **Template matching** (`match.ts`): `findImage(haystack, needle, { threshold?, step? }): Match | null` (pure-TS mean-abs-diff, coarse-to-fine), `locateOnScreen(needle, options?): Match | null` (captures the screen, finds the needle, returns ABSOLUTE coords ready to click), `findAllImages(haystack, needle, { threshold?, step?, maxResults? }): Match[]` (EVERY occurrence, not just the first ÔÇö non-max-suppressed, descending score), `locateAllOnScreen(needle, options?): Match[]` (the screen variant ÔÇö every hit in ABSOLUTE coords), `locateColor(rgb, tolerance?, region?): { x, y } | null` (the first pixel of a color ÔÇö the cheap canvas/game probe, 'find the next red pixel'). `interface Match { x, y, score }`. - **Native window introspection** (`spy.ts`): `windowTree(hWnd, maxDepth?): NativeWindow` ÔÇö the raw HWND hierarchy (class, text, control id, decoded `WS_*`/`WS_EX_*` styles, rect); `renderWindowTree(node): string`; `windowStyles(hWnd)`. The Spy++ / Winspector view; complements UIA by exposing the native structure and the classic-Win32 controls UIA misses. `interface NativeWindow { hWnd, className, text, controlId, styles, exStyles, rect, children }`.
- **Clipboard** (`clipboard.ts`): `readClipboard(): string`, `writeClipboard(text): boolean`, `paste(text): void` (set the clipboard + Ctrl+V ÔÇö the reliable large-text path, no per-keystroke SendInput corruption), `await copy(): Promise<string>` (Ctrl+C + read ÔÇö pull the selected text from any app, even one with no a11y tree). CF_UNICODETEXT via the Global* heap.

## Gotchas (the traps ledger)

- **Selectors are client-scoped, never from the desktop root.** `attach` a window first; `find` defaults to `TreeScope_Descendants` of that window.
- **`ElementFromHandle` is vtable slot 6** (slot 7 is `ElementFromPoint`). The package has the verified slot table; if you hand-roll `vcall`, regenerate slots from `UIAutomationClient.h` and prove each ÔÇö a wrong slot segfaults.
- **Server-side property conditions work** (the 16-byte VARIANT goes by hidden pointer in the x64 ABI). `nameContains` / RegExp still filter client-side.
- **`type` / `sendKeys` / `click` are dropped on a locked session.** Prefer `setValue` / `invoke`; they work locked. `screenshot` (PrintWindow, and the WGC fallback) can come back blank on a locked session ÔÇö rely on UIA reads + `invoke`/`setValue` there, not pixels.
- **Cursor-free is the robust default.** `invoke` / `setValue` / `toggle` and `umbriel.postClick(x, y)` move no real cursor and work on a locked session; the computer-use `dispatch` is cursor-free first (resolve the point ÔåÆ `invoke`). SendInput (`type` / `click` / `sendKeys`) needs an unlocked, **foregrounded** window and can be refused by the foreground lock (e.g. over RDP) ÔÇö `activate()` the window first and prefer patterns.
- **Process is PER-MONITOR DPI-aware** after `initialize()` (shcore `SetProcessDpiAwareness`, falling back to the user32 system-aware call) so UIA bounds and click_point/SendInput coordinates share one physical-pixel space across EVERY monitor ÔÇö including a mixed-DPI multi-monitor desktop, where system-DPI awareness would virtualize a secondary monitor's coordinates and clicks would land wrong. `click()` uses `SetCursorPos` + physical pixels.
- **`SendInput` cbSize must be 40** (handled internally) ÔÇö the x64 `INPUT` is 40 bytes.
- **BSTR names are bulk-copied before free**; never read after `SysFreeString`.
- **A null/zero interface pointer throws, not crashes.** `vcall` guards `thisPtr === 0n` (so `new Element(0n).name` raises a catchable Error instead of segfaulting); MSAA `accChildCount` is clamped so a hostile provider can't force a giant allocation. The COM vtable SLOTs of every engine (UIA + WGC + MSAA/D3D11 + OCR's 16 WinRT slots) are gated by `slot-gate.test.ts` against the SDK headers ÔÇö a transposed slot fails the gate instead of segfaulting at runtime. `example/benchmark.ts` is the reproducible perf artifact (run it before a release): it launches Calculator and prints the published hot-path numbers (single cross-process property read, naive-vs-cached subtree walk, agent-grounding tree build) plus per-path DFG-compilation, so you eyeball a regression against the README table on the same machine ÔÇö it is a measurement run, not a CI ceiling gate (the numbers are machine-specific; do not copy them, regenerate them).
- **Only proven patterns ship.** Calling a pattern an element doesn't support throws a clear message (pointing at `.click()` / `.type()` where relevant). Container `scroll`/`setScrollPercent` (ScrollPattern) is proven; `scrollIntoView` (ScrollItemPattern) is implemented.
- **Browsers / Electron web content IS a UIA tree ÔÇö reach it via `webRoots()`.** Chromium/Edge/Electron render their page into a `Chrome_RenderWidgetHostHWND` child fragment the top-level walk does NOT bridge, so a plain `attach`+`snapshot` of the top-level shows only the chrome ÔÇö but `window.webRoots()` attaches to each render-widget child and `snapshot(window, { extraRoots: webRoots })` (exactly what the MCP `desktop_snapshot` does automatically) splices that page DOM into the tree, so you read / `setValue` / `invoke` web controls with the same refs and patterns as any native control ÔÇö cursor-free, no foreground, no screen-reader flag (proven in `example/web-content.integration.test.ts`: read an input, set it, invoke a button, read the DOM update). **Scrolling** a Chromium page is also cursor-free: its UIA ScrollPattern FALSELY reports `verticallyScrollable:false` even on a tall page (so a ScrollPattern position jump silently no-ops ÔÇö `scroll {to/top/bottom/page-up/page-down}` correctly refuses it and steers you to a directional scroll), but a directional `scroll {ref, direction:down}` posts a `WM_MOUSEWHEEL` to the page's Chromium host window (`Element.chromiumHostHandle()`) and the page actually moves, no cursor / foreground (proven in `example/chromium-scroll.integration.test.ts`). The pixel layer (`captureScreen` + `locateOnScreen` + coordinate `click`) is the fallback ONLY for surfaces with no a11y tree at all (games, canvas/WebGL, custom-draw). WebView2-embedded content ÔåÆ CDP (`--remote-debugging-port`).
- **UWP / WinUI Store apps read via UIA ÔÇö they are NOT blind.** Settings, Store, Photos, Mail, Phone Link, etc. host their XAML in a `Windows.UI.Core.CoreWindow` under an `ApplicationFrameWindow`; UIA bridges them, so `attach` + `desktop_snapshot` on the frame yields the controls cursor-free (proven: Settings' full navigation, 26 refs / 14 list items ÔÇö `example/uwp-content.integration.test.ts`).
- **Qt (PySide6 / PyQt6 / PyQt5) reads + drives via UIA ÔÇö a pinned first-class toolkit.** Qt powers a tier-1 slice of the desktop (OBS, VLC, Telegram, KeePass, qBittorrent, Wireshark, the whole KDE stack); a Qt `QWidget` window bridges its widgets to UIA, so `attach` + `desktop_snapshot` yields the `Button`/`Edit`/`CheckBox`/ÔÇĞ as actionable refs and `set_value`/`invoke`/`toggle` drive them cursor-free (proven: `theQtButton`/`theQtEdit`/`Enable Qt Thing` read + a ValuePattern setValue round-trips ÔÇö `example/qt-widget-tree.integration.test.ts`). The bridge is version-fragile (Qt's QML/Qt-Quick path may instead expose nothing ÔÇö fall back to `ocr`/`click_text` there), which is exactly why the regression test pins it.
- **Java Swing/AWT is drivable via `java_tree` (the Java Access Bridge) ÔÇö NOT a dead end.** A Java AWT/Swing window (class `SunAwtFrame`/`SunAwtDialog`) exposes ONLY its bare frame to BOTH UIA and MSAA ÔÇö never one piece of app content ÔÇö because the JVM speaks the separate Java Access Bridge protocol, not UIA. `java_tree {hWnd}` reads the REAL tree (role/name/states/bounds) over the Access Bridge, cursor-free / background. To ACT, `java_invoke {hWnd, name}` presses a button / toggles a check box / fires a menu item and `java_set_text {hWnd, name, text}` types into a field ÔÇö both cursor-free / background via the bridge (doAccessibleActions / setTextContents), no cursor or foreground; `click_point` at a node's bounds center is the pixel fallback. The MCP `attach` note and a cold (0-control) `desktop_snapshot` of a `SunAwt*` window both steer you to `java_tree`. It returns null (with a hint) if the JVM has not loaded the bridge ÔÇö enable it with `jabswitch -enable` then RESTART the app, or launch the app with `-Djavax.accessibility.assistive_technologies=com.sun.java.accessibility.AccessBridge`. Only when the bridge is unavailable does `screen_capture` + `ocr`/`click_text` remain the fallback. (JavaFX reaches the bridge only when initialized with assistive tech active ÔÇö where it does, `java_tree` reads it; otherwise it may instead expose a UIA tree.)
- **The a11y tree is LAZY ÔÇö a cold/idle window can read sparse.** Like Chromium, a UWP/WinUI app (and a backgrounded browser) builds its UIA tree on demand and tears it down when long-idle, so the FIRST `desktop_snapshot` of a just-launched or long-idle window can be near-empty. Re-snapshot (poll a few times) until it populates; if it stays empty it may need a brief activate. A `0`-control snapshot of an app that clearly has UI usually means a cold tree, not a tree-less surface ÔÇö don't fall back to pixels until a re-snapshot also comes back empty.
- **Menus are drivable cursor-free ÔÇö expand, then re-snapshot.** `expand()` the top `MenuItem` (ExpandCollapse pattern ÔÇö no focus, no cursor), then `desktop_snapshot` again: a modern (WinUI/Win32) app surfaces the submenu's `MenuItem`s in the SAME tree (e.g. Notepad File: 4 ÔåÆ 17 items), so `invoke()` the one you want. A WinUI/UWP combobox renders its dropdown items into the SAME tree (as ref'd `ListItem`s under the combobox) after `expand` ÔÇö there is a brief render race, so the MCP `expand` / `find_and_act {do:expand}` SETTLE (poll the re-snapshot until the items appear) and the revealed items are already in the returned snapshot; just `invoke` the one you want (no own-window hunt). A CLASSIC Win32 combobox (its `ComboLBox` list), a classic `#32768` context menu, and a WinUI 3 desktop flyout / autocomplete list instead open in their OWN untitled top-level window: `expand`/`context_menu` auto-return that popup's hWnd (or `list_windows {includePopups}` ÔÇö SDK `listWindows({includeUntitled})` ÔÇö surfaces it), then `attach` it by hWnd and snapshot its items (a Chromium dropdown's items are in its webRoots, like any Chromium window). A right-click CONTEXT menu opens cursor-free via the UIA COM path ÔÇö `context_menu {ref}` (Element `showContextMenu()` ÔåÆ IUIAutomationElement3::ShowContextMenu); it is **provider-dependent** (works on e.g. the taskbar / shell items, not every control), and when it works it returns the menu's untitled popup hWnd to `attach` + drive. The POSTED-message route does NOT work (a posted right-click / WM_CONTEXTMENU / the Apps key never raises the menu ÔÇö TrackPopupMenu needs real input-thread state, verified). If `context_menu` reports no menu appeared, fall back to a real-cursor right-click (`click {cursor:true, button:'right'}`, needs a foregrounded desktop) then `list_windows {includePopups}`, or drive the control's `expand`/SplitButton pattern or its keyboard accelerator.
- **Non-English / different builds** differ: Calculator's result element is `automationId: 'CalculatorResults'`, buttons are named `Five`/`Plus`/ÔÇĞ (full words). Configure selectors per app.
- **Threading / events:** v1 is STA, fire-and-forget out-of-process driving. WINDOW and PROCESS lifecycle events ARE available (`wait_for_window` via `SetWinEventHook` pumped on the main thread ÔÇö no foreign-thread hazard; `wait_for_process` polls a toolhelp32 snapshot). UIA property/structure event SUBSCRIPTION and self-UI automation need MTA and remain out of scope ÔÇö poll with `wait_for` / `wait_idle`.
