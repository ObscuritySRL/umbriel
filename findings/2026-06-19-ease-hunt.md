# Ease-hunt: make AI Windows automation easier (fewer tokens/steps, more reliable) — 2026-06-19

Owner reframed Panel B: not "invent arbitrary features" but "a tool/approach that makes the AI's job easier —
fewer tokens, faster, fewer steps, more reliable", and "you're the end user — if it helps, add it." Ran a
5-seat panel (token-economy, step-collapse, direct-value-set, reliability, auto-wait), each requiring a live
repro. Verified each candidate myself before acting (the session's rule). Outcome: 3 shipped, 2 declined.

## Shipped (verified live, pushed)

1. **perf(mcp): trim duplicated ref/element schema descriptions** — REF_DESC was inlined into 29 tool schemas,
   ELEMENT_DESC into 23, re-stating the ref/#generation contract that INSTRUCTIONS already carries once and the
   code enforces regardless. Shortened to the load-bearing point-of-use directive. Live-measured: tools/list
   58,059 -> 53,030 chars (~14,515 -> 13,258 tok, -8.7%) on EVERY session before the first tool call. Zero
   behavior change.
2. **feat(mcp): toggle {state}** — optional {state:true|false} sets a checkbox/switch to a known state
   idempotently (no flip if already there), collapsing "ensure checked" from 3 steps (inspect -> branch ->
   conditional toggle) to ONE call and making it retry-safe. Omitting {state} = today's flip (back-compat).
   Verified live on a synthetic BS_AUTOCHECKBOX over the MCP wire (ground truth via BM_GETCHECK). New test.
3. **feat(mcp): find_and_act auto-waits for ENABLED** — it already auto-waited for a not-yet-present selector
   but instant-threw on a present-but-DISABLED control, forcing a 3-call dance (refusal -> wait_for -> re-act)
   on the most common act race. Folded the enabled-wait into the existing existence-poll loop, sharing the
   {timeout} budget. timeout:0 stays instant. Verified live on Calculator's never-enabling memory button
   (timeout:1200 waits ~1238ms before refusing, vs instant at timeout:0). Updated description + steer + test.

## Declined (NOT shipped — honest reasons; candidates for the owner)

4. **UWP cold-tree warm-up on attach** — broaden rebuildSnapshot's chromium-only warm-up gate (mcp.ts:556,
   `if (chromium && marks < SNAPSHOT_COLD_REFS)`) to also cover a sparse WinUI/UWP tree (a className predicate
   for Windows.UI.Core.CoreWindow / ApplicationFrameWindow / WinUIDesktopWin32WindowClass / Microsoft.UI.Content.*).
   The cold-UWP-attach case IS real per the maintainer's own example/epoch-refs test (it loops 8x400ms "until
   Five appears" after a bare attach of a UWP Calculator). BUT: I could NOT reproduce it (`.scratch/probe-cold-uwp.ts`:
   Calculator fresh=50 / minimized=50 refs, Settings fresh=34 / minimized=36 — all WARM, even minimized). So I
   cannot verify (a) that the fix triggers on a real cold tree here, nor (b) that re-querying actually WARMS a
   cold UWP CoreWindow tree the way it provably warms a chromium one (different on-demand mechanism). And it adds
   ~200ms to every genuinely-small WinUI attach (the warm-up loop breaks after one non-growing re-query, so the
   cost is bounded — but it is non-zero for a real benefit I can't demonstrate). REVISIT when a cold UWP attach
   reproduces: confirm re-query warms the CoreWindow tree, then gate `(chromium || winui)`.

5. **Own-window popup auto-attach inline** — when expand/invoke opens a dropdown/menu/flyout in its OWN window,
   auto-attach + render its items inline to save the manual `attach {hWnd}` round-trip (3 calls -> 2 for a menu
   pick). NOT a clean surgical change: to let the agent act on popup items BY REF, the popup must become the
   attached `current` snapshot (refs resolve against `current`) — which re-points `attached` to a TRANSIENT
   window that closes after the pick, leaving the agent's NEXT action pointed at a dead window (a new friction
   replacing the old one). Unlike wait_for_window {attach:true} (which attaches a PERSISTENT dialog the agent
   keeps driving), a dropdown popup is ephemeral. Doing this right needs a design decision (auto-restore
   `attached` to the parent after the popup closes, or a separate popup-ref namespace), not a surgical edit.

## Note

The 3 shipped are the clean, verifiable wins. The 2 declined are real ideas that need either a reproducible
cold-UWP case (4) or a lifecycle design decision (5) — both the owner's call, recorded here so a future hunt
doesn't re-chase them blindly. Verify-before-ship held: I built only what I could prove helps.
